# SG-CoT: An Ambiguity-Aware Robotic Planning Framework using Scene Graph Representations 

Akshat Rana, Peeyush Agarwal, K.P.S. Rana, Amarjit Malhotra Netaji Subhas University of Technology 

Abstract —Ambiguity poses a major challenge to large language models (LLMs) used as robotic planners. In this letter, we present Scene Graph-Chain-of-Thought (SG-CoT), a two-stage framework where LLMs iteratively query a scene graph rep-resentation of the environment to detect and clarify ambiguities. First, a structured scene graph representation of the environment is constructed from input observations, capturing objects, their attributes, and relationships with other objects. Second, the LLM is equipped with retrieval functions to query portions of the scene graph that are relevant to the provided instruction. This grounds the reasoning process of the LLM in the observation, increasing the reliability of robotic planners under ambiguous situations. SG-CoT also allows the LLM to identify the source of ambiguity and pose a relevant disambiguation question to the user or another robot. Extensive experimentation demonstrates that SG-CoT consistently outperforms prior methods, with a minimum of 10% improvement in question accuracy and a minimum success rate increase of 4% in single-agent and 15% in multi-agent environments, validating its effectiveness for more generalizable robot planning. 

Index Terms —AI-Enabled Robotics, Human-Centered Automa-tion, Human-Robot Collaboration. 

I. I NTRODUCTION 

# LARGE Language Models (LLMs) have transformed the landscape of Artificial Intelligence, providing impressive generalization across domains. Naturally, this progress has also been extended to robotics, where LLMs have been used to achieve reliable human-robot interaction for tasks such as manipulation [1], navigation [2], [3] and planning [4], [5]. However, the robots may face ambiguous situations arising from either the state of the environment or underspecification by the user itself. For example, an instruction such as ”Pick up the cup” becomes ambiguous if there are multiple cups or no cups in the environment, while an instruction like ”Get me something to eat” is ambiguous because it may lead to multiple valid interpretations. In such cases, a reliable agent must learn to ask for clarification from the user or another robot performing the same task as it may otherwise lead the robot into taking suboptimal or unsafe actions [6], [7]. Since LLMs are inherently overconfident and are not explic-itly trained to handle ambiguity [8], it becomes a fundamental challenge for AI-Enabled robotics. Previous approaches [6], [9] estimate the uncertainty levels in the responses of LLMs to detect ambiguity. However, their evaluation typically focuses on ambiguities in the instruction instead of ambiguities induced by the state of the environment. They also rely on vision-language models (VLMs) like ViLD [10] or BLIP [11] for perception. Representations generated by these VLMs are often just a list of objects or a high-level description, making it hard for them to disambiguate complex instructions referring to a specific attribute (e.g., ”Get me the object with green buttons” )or a relation among objects (e.g., ”Get me the object to the left of the apple” ). To bridge the above mentioned gap, a Scene Graph-Chain-of-Thought (SG-CoT) is proposed in this letter. SG-CoT is a robotic planning framework that integrates scene graphs with the reasoning capabilities of LLMs to enhance robustness against ambiguity. The framework works in two stages. First, a structured scene graph of the environment is built using a VLM capturing semantic attributes and relationships among objects. This representation moves beyond high-level descriptions and reveals environmental nuances, which are crucial for detecting ambiguities in various scenarios. Secondly, the LLM is paired with a tool-use capability. The LLM receives the high-level descriptions of its observations and the instruction as input. It then begins its thought process and calls a valid API function to access nodes and edges of the scene graph, required to complete the instruction, grounding its thoughts into visual context. Upon retrieval of all relevant information, the LLM makes a decision on whether multiple valid interpretations of the user instruction exist or not. In this way, ambiguities arising from both the environment and underspecification by the user can be resolved. In contrast to purely text-based planners, SG-CoT provides integration of scene graphs with iterative reasoning, allowing the LLM to pinpoint the reason for the ambiguity and hence generate clarification questions targeted towards resolving the ambiguity. Two experiments were conducted to validate the proposed method. First, evaluation was performed on simulated tabletop environments presented by SayCan [4]. Second, the methodol-ogy was tested on multi-robot tasks derived from the LEMMA benchmark [12]. To these environments, ambiguity across three different categories was introduced by applying perturbations to both the environment states and the instructions, removing contextual information from the robot’s view. Based on the presented results, it was observed that SG-CoT demonstrated superior performance compared to previous approaches in both executing tasks and asking the correct clarification questions whenever required. Task planning with LLMs: Robotic task planning [13] in-volves translating natural language commands into executable action sequences based on input and past observations. There have been several previous approaches [4], [5], [9], [14]– [17] that use LLMs for this purpose. To increase reliability of LLMs as planners, ProgPrompt [14] used a programmatic prompt structure, LLM-Planner [16] used a framework for dynamic grounded re-planning and ReAct [18] interleaved reasoning traces with actions. To further enhance reliability, LLMs employed feedback loops from the environment [5], [19], [20] to detect failures and iteratively refine their plans. However, these approaches assume the environment contains sufficient context to infer a definitive action and will execute an action regardless of the state of the environment. As mentioned in KnowNo [6], such behavior can cause actions with clear safety hazards, like placing a metal bowl in the microwave. 

Ambiguity resolution in LLMs: Several prior works have proposed methodologies to tackle the inability of LLMs to handle ambiguity [21]–[24]. For robotics in particular, most previous research focuses on underspecification by the users. KnowNo [6] used conformal prediction to generate a statis-tically guaranteed set of plausible actions, asking for help when this set contains more than one option. CLARA [9], which classifies user commands as clear, ambiguous, or in-feasible through uncertainty estimated via stochastic context sampling, is used as the basis for the problem formulation, with focus restricted exclusively to ambiguity. Furthermore, the problem is expanded beyond simple user underspecification to address a broader class of ambiguities: those arising from the environment itself, the inherent uncertainty of partially observable environments, and those occurring in more complex instructions. 

Scene graphs: A scene graph is a semantic representation of a scene in which the environment is modeled as a set of objects, their attributes, and the relationships between them [25]. They are applied in computer vision for image retrieval [26], image captioning [27], [28], visual question answering [29] and image generation [30]. Various works have also ex-plored their application in robotics [31]–[33] and more recent approaches have integrated them with LLMs for generation and understanding [34]–[39]. The proposed framework is built upon graph-grounded LLM planners like SayPlan [38], that pairs an LLM with an API allowing it to expand and collapse a hierarchical 3D scene graph to dynamically manage its own context. However, unlike SG-CoT, SayPlan does not ask for clarification, making it less reliable in ambiguous situations. In addition, it assumes a pre-constructed scene graph and does not support incorporation of open-vocabulary representations of the graph structure. Based on the above mentioned three components, this letter makes three primary contributions. First, it proposes SG-CoT, a framework that integrates scene graph construction with iterative chain-of-thought reasoning by an LLM to enhance robotic task planning under ambiguous situations. Secondly, ambiguous scenarios are formally classified into three distinct types, including environment-induced ambiguities, which are largely unexplored by prior approaches. Third, an evaluation metric is introduced to measure correctness of clarification questions generated. III. P ROPOSED METHODOLOGY 

In this section, the proposed framework, referred to as SG-CoT, is introduced. The problem statement is presented first, followed by an introduction of scene graph generation and representation, and finally, the reasoning process that utilizes the scene graph for disambiguation and interaction. 

A. Problem Statement 

The problem setup involves a single or multi-agent robotic system integrated with an LLM as a high-level planner for each individual robotic agent Ri. Given a natural language instruction I and an image observation O from the environ-ment, the objective of the system is to generate a sequence of high-level actions ai ∈ A , where A denotes the predefined set of executable actions available to the robot, including asking for clarification, to accomplish the defined task. To evaluate the planner’s robustness in handling ambiguity, controlled perturbations were introduced in both the instruction and the environment, as shown in Fig. 1. Instruction-level ambiguity, referred to as underspecification, involves omission of ex-plicit referential grounding in the instruction. For example, replacing a grounded reference (e.g., ”place the red block over the green bowl” ) with an ungrounded term (e.g., ”place something over the green bowl” ). Environment-level ambiguity relates I to the set of objects X present in O. Let A(I)

be the set of attributes describing a target object referenced in I, A(o) be the set of attributes of an object o and let 

Xmatch = {o ∈ Xscene | A(o) ⊇ A(I)} be the set of objects in the scene satisfying these properties. Absence is defined as the condition Xmatch = ∅, and multiplicity as the condition |Xmatch | > 1. Multi-agent systems are also subject to observation-level ambiguities occurring from the partially observed nature of the environment. B. Scene Graph Generation 

A scene graph provides a structured representation of the en-vironment, derived from the observation O. Formally, the scene graph is defined as G = ( V, E ), where V = {v1, v 2, . . . , v n}

represents the set of nodes, each corresponding to an object 

o ∈ Xscene detected in the environment. Each node vi is associated with a set of attributes A(vi) = {a1, a 2, . . . , a k}

that describe its observable properties, such as color, size, or higher-level states like is_broken or is_open . The edge set E captures pairwise relations between objects, where each edge is represented as a tuple (vi, r ij , v j ) ∈ E, where rij 

denotes a semantic relation between vi and vj generated from an open-ended vocabulary. Fig. 1: The figure shows possible configurations of the defined ambiguity types. For each category, the image of the left is the global perspective and the human instruction, and the one on the right is the egocentric view of the robot and its clarification question. A prompting-based method was used for scene graph gener-ation, bypassing the need for dedicated training. To construct the scene graph, first the graph nodes were instantiated using a list of detected objects from an open-set object detector like Grounding DINO [40] or a VLM. To build the edges, the semantic reasoning capabilities of a VLM are utilized to generate attributes and relationships from an open-vocabulary. Prompting the VLM for all object pairs requires O(n2)

computational complexity, so instead, following SG-Nav [39], the VLM was prompted with the set of objects to jointly generate their attributes and relationships in one shot, reducing the computational cost. The VLM was constrained to strictly output a JSON object, which enables deterministic parsing by the retrieval functions. 

C. Reasoning Process 

Once the scene graph G is constructed, the LLM follows an iterative reasoning framework allowing it to utilize the graph effectively for task completion. Due to its single-pass and sequential generation process, chain-of-thought prompting [41] is heavily dependent on early decisions, allowing initial errors to propagate and cause errors in the final answer [42]. In contrast, the core of SG-CoT is an iterative, multi-turn reasoning process that enables the LLM to actively query the scene graph, ground its thought process, and refine its understanding before committing to an action. We hypothesize that iteratively grounding the LLM’s thought process in reality can mitigate accumulation of previous errors and subsequent hallucination. In addition, the LLM is able to pinpoint the root cause of the ambiguity, allowing it to ask a relevant clarification question that will help in disambiguating the situation. This loop ensures that each reasoning step is validated with the scene graph, preventing propagation of hallucinations. The process is initialized by constructing an initial prompt P0, which contains the user instruction I, the set of detected objects, and the schema of the scene graph. The schema includes the set of all valid attribute types (e.g., color , size ) and edge types (e.g., 

on_top_of , inside_of ), providing the LLM with the necessary vocabulary for querying the graph. A conversation history, H, is maintained and is initially set to this prompt, 

P0.Following initialization, the process enters an iterative loop where at each turn t, the LLM is prompted with the accu-mulated history Ht. It generates a response consisting of two parts. First, a reasoning trace, Tt, which details the step-by-step articulation of its reasoning. Secondly, if the LLM concludes that more information is needed from the scene graph, it ends its turn by outputting a call to a retrieval function, ft. Finally, if the LLM has enough context to make a decision, it returns an action sequence a.The system intercepts ft from the response generated by the LLM. The function queries G and retrieves the relevant nodes or edges. The LLM has access to two different retrieval func-tions, retrieve_node and retrieve_edge . The first function, retrieve_node , extracts a list of objects from the graph based on some condition. It can be either a direct object name or by a specific attribute key-value pair. For exam-ple, a query like retrieve_node(attr_key=’color’, attr_val= ’red’) would return the names of all nodes in the scene that have the color red. The second function, retrieve_edge , can be used to query the spatial and semantic relationships between nodes. It accepts optional arguments for the source node, the target 

node, and the relation type. These parameters can be used individually or in combination to filter the scene’s edge list. For example, retrieve_edge(source=’blue bowl’, relation=’above’) would return a list of strings detail-ing all the objects the ’blue bowl’ is ’above’, such as ”the blue bowl is above the red block”. After the function call is made, it returns the results Rt.The conversation history is updated according to the rule 

Ht+1 = Ht ⊕ (Tt ⊕ ft ⊕ Rt), where ⊕ denotes concatenation. The LLM is then prompted with the augmented history, Ht+1 

through which it can review its previous thought process with the retrieval results and make inferences about the environ-ment. The loop of reasoning continues until the termination con-dition is met. This occurs when the LLM’s thought process in Tt concludes that it has sufficient information to take a definitive action. Alternatively, the process terminates if the model reaches a predefined limit of max turns rounds of thought, after which it is prompted to conclude its reasoning. At this point, instead of calling another retrieval function, it generates a final action sequence, A = ( a1, a 2, . . . , a k).This sequence can consist of physical actions or a clarification question to resolve an identified ambiguity. A. Evaluation Metrics 

Each baseline was evaluated over the following two metrics.  

> •

Success Rate (SR): This metric defines how often the model produces the desired action sequence. For un-ambiguous situations SR is 1 only if the correct action sequence is executed and 0 otherwise. For ambiguous, SR 

is 1 when the robot asks for clarification, and 0 otherwise.  

> •

Correct Question Rate (CQR): This metric was used to check whether the model asks for the right question and understands the source of ambiguity. To check this, the LLM is prompted to output a tag along with the question, specifying its intent. CQR is 1 when the tag matches the ground truth ambiguity, and 0 otherwise.

To incorporate the ambiguity types described earlier, the environment and instructions were modified as follows. Un-derspecification was introduced by replacing the target object in the instruction with a term drawn from a curated dic-tionary of vague words. For absence, the target object was removed, and for multiplicity the target object was duplicated. In total, 400 trials were conducted with 100 randomized configurations per ambiguity category, and 100 clear scenarios. In each trial, the LLM received the current observation and the task instruction. The action space consisted of two high-level actions: pick_and_place and ask , which together define a planning interface over the environment state. The action pick_and_place expects two arguments, the object to be picked and the object where it should be placed, and is executed on the precondition that no object is currently held, resulting in the object being placed inside or on the specified receptacle. The action ask expects a question string and an ambiguity tag defining the type of ambiguity, and is invoked when the observation does not allow a unique grounding of the instruction. The goal condition for each unambiguous trial is satisfied when the intended target object is successfully placed in the specified container or spatial location. For ambiguous trials, the goal condition is considered satisfied if the correct clarification question is generated, as an assumption is made that once the appropriate clarification is obtained from the user, the ambiguity is resolved and the instruction can be successfully executed. 

Results: As shown in Table II, instruction-level ambiguity was the most manageable for all previous approaches with an average SR of 77% compared to 56.6% for environment-level ambiguities as they are not explicitly visible from the instruction alone. SG-CoT maintained high performance across both settings, achieving SR of 99.0% and 77% un-der underspecified instructions and 86.5% and 80.5% for environment-level ambiguities with gemini-2.5-flash 

and Qwen3-VL-2B-Instruct respectively. The notable increase in CQR by 10% with gemini-2.5-flash and 11% with Qwen3-VL-2B-Instruct obtained by SG-CoT over the previous best Inner Monologue, supports the hypoth-esis that selectively retrieving only the environment elements relevant to task completion enables the model to formulate accurate clarification questions. There is a performance drop in multiplicity scenarios as compared to the other cases, as it still allows the LLM to take an action that matches the user request, unlike absence where picking up the target block would be inherently invalid. For example, if two valid red blocks are present, the model may arbitrarily pick one to complete the user instruction, ignoring the ambiguity. In contrast, If the scene graph does not return a block, the ambiguity is relatively explicit and easier for the model to detect. The evaluation metrics were consistently higher for gemini-2.5-flash 

compared to Qwen3-VL-2B-Instruct , suggesting that performance scales positively with model capacity. Overall, SG-CoT achieves consistently superior performance across both evaluation metrics and LLMs compared to prior ap-proaches. 

Ablation: An ablation study was conducted to analyze the contribution of individual components in the framework. In the first ablation, instead of iteratively retrieving relevant parts of the scene graph, the model is provided with the entire scene graph directly, based on which it must predict and action sequence. This results in an overall drop of 0.34/0.38 (SR/CQR) for Qwen3-VL-2B-Instruct , and 0.32/0.39 for 

gemini-2.5-flash , demonstrating the role of iterative reasoning. Restricting the model to a single retrieval step before removing the scene graph access further leads to a lower decrease of 0.18/0.20 for Qwen3-VL-2B-Instruct ,and 0.17/0.25 for gemini-2.5-flash . For the final abla-tion, the scene graph is built without the edges, resulting in reductions of 0.17/0.23 for Qwen3-VL-2B-Instruct , and 0.17/0.34 for gemini-2.5-flash , showcasing the role of scene graph accuracy. 

Computational Cost and Scalability: As shown in Table III, due to the history concatenation and the additional scene graph generation step, SG-CoT does require higher token usage and LLM calls. The latency also scales approximately linearly with the total number of tokens processed and the number of calls. However, increasing the total number of objects in the environment does not proportionally increase the number of reasoning iterations, since scene graph generation is performed in a single pass and subsequent retrieval depth depends upon the size of the user request, as the model only retrieves parts of the graph relevant to the user request, supporting scalability with increase in environment size. 

C. Partial Observability 

This section entails the setup and the performance of the proposed method over the final ambiguity type caused by partial observability in multi-agent environments, with a focus on inter-agent communication for resolving observation-level ambiguity. 

Experimental Setup: To evaluate the performance of SG-CoT in handling observation-level ambiguity within multi-agent systems, the same environment was utilized as before but the tasks were adapted from the LEMMA [12] benchmark. Specifically, the proposed method was assessed on the Stack and Pass tasks introduced in their benchmark. The environment consists of two cooperative robotic agents, each operating under partial observability with access to a distinct individual workspace as well as a shared workspace. Each robot can per-ceive only the objects within its own and the shared workspace, thereby requiring communication with the other agent to obtain missing observations needed for task completion. As illustrated in Fig. 1, the work space was divided into three regions, red being the workspace for first robot, blue for the second robot, and the green area serves as the shared workspace. The input of each robot was occluded to introduce the partial observability. Furthermore, the action space was expanded by introducing a new clarification action, ask_robot . This action enables an agent to query the other robot regarding its local observations. If the target objects required for task completion are not present in the robot’s view the robot must ask for observations of the other robot and use the shared workspace for transfer of objects. Since there is only one ambiguity type, only SR was calculated for this experiment. For a clear scenario, SR is set to 1 for execution of the correct pick_and_place action. If the object to be picked up is occluded, SR is 1 for the 

ask_robot action, and if only the place object is not visible, 

SR is 1 for the robot to put the target object over the shared space for the other robot to pick up.

SG-CoT forces an LLM to query parts of the scene graph for details of the environment, preventing it from making up new details and rely solely on the returned results for its reasoning. V. C ONCLUSION 

In this paper, Scene Graph-Chain-of-Thought (SG-CoT) was proposed, a framework that allows large language model (LLM) based robotic planners to iteratively query and reason over a scene graph representation of the environment in order to make decisions. This framework allows the LLM to be resis-tant to ambiguity by grounding its reasoning in reality, shown by a 4% enhancement in overall success rate in single-agent environments and 15% in multi-agent settings. The method also allows the LLM to pinpoint the reason of ambiguity and ask a question required for disambiguation, leading to a minimum 10% improvement in the correct question rate metric. However, there are some limitations to SG-CoT. The method relies on a vision–language model (VLM) to infer spatial and semantic relationships in a single step, which can hallucinate with real-world objects and introduce incorrect edges into the scene graph, potentially hampering the reasoning process. In addition, the token usage and inference latency increase with the number of ambiguities present in the scene, which may limit scalability for more complex instructions. Finally, the current evaluation is limited to simulation-based experiments, and no real-world experiments were performed.
